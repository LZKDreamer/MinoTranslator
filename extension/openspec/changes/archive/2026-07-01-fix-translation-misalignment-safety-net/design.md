## Context

`fix-subtitle-pipeline-alignment` 完成后，断句流水线为：`parseJson3ToWords` → `preSegmentPhraseEvents`（按 `\n-` 说话人 / 按内部 `.?!。？！` 标点预切）→ `segmentSentences`（主循环切句 → 稀疏垃圾 → 向后合并 → 前向合并 → 过短句合并 → 长句切分 → 重叠截断/最小展示/短句碎片合并/重复幻觉丢弃）→ `cleanCueText` → `batchTranslateSentences`。

翻译侧 prompt 明确要求 AI："Do NOT merge lines, do NOT skip lines, do NOT reorder lines"，输出 JSON 对象 `{"0":"...","1":"..."}`，长度严格等于输入。这个约束的**前提条件**是：每条输入原文是语义完整的句子。一旦送了半句话，AI 为了产出可读译文会合并相邻半句，导致输出长度 < 输入长度，`normalizeTranslationArray` 补的空字符串位置错位，后续全批错位。

日志分析（`subtitle-pipeline-log (3).txt` 299 句）确认：
- 24 个句子不以 `.?!。？！` 结尾（占 8%）。
- `#56`（8.2s）+`#57`（6.6s）合并 14.8s 触发长句切分，在最大间隔处切，左半无标点 → AI 合并 → #57 起全批错位。
- 其余 22 个无标点句 < 12s，由主循环的大间隔切句、说话人切句或预切分残余产生，未触发长句切分但仍可能诱导 AI 合并。

约束：
- 保持"本地断句 + AI 只翻译"分工，不引入 AI 端断句。
- 不重写 `segmentSentences` 主循环（53 个测试依赖现有结构）。
- 必须用 `f (2).txt` + `subtitle-pipeline-log (3).txt` 回归。
- 上线标准 = 翻译不错位 = 不送半句话给 AI。

## Goals / Non-Goals

**Goals**
- 最终送 AI 的每条句子以句末标点结尾（极端无标点视频除外）。
- 翻译输出数组长度 = 输入数组长度，不错位。
- `#56`+`#57` 不再被切开；24 个无标点结尾 case 降到接近 0。
- 长句切分优先在语义边界（标点）切，不在纯时间间隔处切。
- 改动小、风险低、不破坏现有 53 个测试。

**Non-Goals**
- 不重写主循环，不改合并阈值，不改预切分，不改渲染器，不改 prompt。
- 不处理 #78/#79 空翻译（翻译侧问题，另立 change）。
- 不追求每个翻译单元 ≤4s/≤70 字符（"跟得上嘴"是软约束，由渲染层分行和 AI 译文简洁性保证；"对位+完整"才是硬约束）。
- 不消除所有无标点结尾（连续 3+ 句无标点且合并超 15s 的极端 case 仍可能残留，接受并以 WARN 日志监控）。

## Decisions

### D1：安全网放在 `segmentSentences` 末尾、`return sentences` 前
**选择**：在所有现有后处理（重叠截断、最小展示、短句碎片合并、重复幻觉丢弃）完成之后，新增一个独立的后处理循环执行"无标点前向合并"。

**理由**：
- 安全网必须看到最终形态的句子，才能判断"这条是否半句话"。
- 放在最后保证：不管上游哪个步骤切错（主循环、长句切分、说话人、合并反向拆分），统一兜底。
- 独立循环不侵入现有逻辑，53 个测试的中间步骤行为不变，只改最终输出。

**备选**：
- (A) 在主循环内累积到标点才 emit。拒绝：等于重写主循环，风险高，破坏 ghost/说话人/lineBreak 处理。
- (B) 在每个切断点单独加"半句话检查"。拒绝：切断点有 4+ 个，逐个加检查复杂且易漏，新切断点会忘加。

### D2：安全网合并规则——无标点 + gap<1s + 合并<15s → 前向合并
**选择**：
```
for i in 0..n-2:
  if not sentences[i].text 以 .?!。？！ 结尾:
    gap = sentences[i+1].start - sentences[i].end
    mergedDur = sentences[i+1].end - sentences[i].start
    if gap < 1.0 and mergedDur < 15.0 and not hardBreakBetween(i, i+1):
      merge sentences[i] into sentences[i+1]  # 前向：当前并入下一句
      remove sentences[i]
      re-check i  # 合并后新句子可能仍无标点，继续往前看
```

**参数选择**：
- gap < 1s：与现有 `BACKWARD_MERGE_GAP_MS=1000` 对齐。gap>1s 说明语义上确实断了（说话停顿），不合并。
- 合并 < 15s：略高于现有 `MAX_SENTENCE_DURATION_SEC=12`。给安全网留余量——12s 是"主动切分"阈值，15s 是"被动兜底合并"上限，两者不冲突。超 15s 接受半句话（AI 翻译半句比错位好）。
- 前向合并（当前并入下一句）：与现有"前向合并"方向一致，避免引入新方向。半句话通常是上一句的尾巴，并入下一句让它和下一句一起翻，语义更完整。
- `hardBreakBetween`：若两句间有说话人切换或大间隔（>2s）硬切，不合并（保留说话人边界）。

**备选**：
- (A) 后向合并（下一句并入当前）。拒绝：方向与现有前向合并相反，且半句话当头不如当尾自然。
- (B) 不限 gap，纯按"无标点就合并"。拒绝：会把隔了 5s 静默的两句强行合并，破坏语义。

### D3：长句切分改为标点优先
**选择**：`duration > MAX_SENTENCE_DURATION_SEC` 时，切分点选择逻辑改为：
1. 优先找 `sent[q].text` 以 `.?!。？！` 结尾的 q（q < len-1），在该 q 之后切。
2. 若有多个标点切分点，选**最接近中点**的那个（平衡左右半长度）。
3. 若整段无标点，退化到现有"最大间隔"切分。

**理由**：长句切分本就是"硬约束（长度）下的妥协"，妥协点应选在语义边界，否则切出半句话又触发安全网合并，形成"切了又合"的抖动。标点优先让切分结果稳定。

**备选**：
- (A) 长句不切，接受 15s 长句。拒绝：长句超 12s 时渲染层分行困难、AI 译文可能过长难读。
- (B) 标点优先但选第一个标点。拒绝：可能切出 2s+13s 极不平衡的两半，13s 半仍触发再切。

### D4：极端 case 的 WARN 日志
**选择**：安全网循环结束后，扫描最终 sentences，若存在不以 `.?!。？！` 结尾的句子，`debugLog` 输出 `WARN: incomplete sentence remains at #N: "..."`。

**理由**：连续 3+ 句无标点且 gap<1s 合并超 15s 的极端 case（ASR 连漏标点）极少见但可能存在。WARN 不阻断流程，便于线上日志定位异常视频、人工评估是否需调参。

**备选**：直接 throw 中断。拒绝：上线产品不应因单句无标点中断整视频翻译。

## Risks / Trade-offs

- **[安全网合并出超长句]** 合并后可能 14s+，触发渲染层分行压力。**缓解**：15s 上限 + AI prompt 要求译文简洁 + 渲染层已有分行逻辑。14s 句在韩语长对白中可接受。
- **[安全网与长句切分抖动]** 安全网合并出 14s 句 → 长句切分（12s 阈值）可能再切 → 切在标点处（D3）→ 不再是半句话 → 安全网不触发 → 稳定。D3 标点优先消除了抖动。
- **[前向合并方向误判]** 英语里 "Oh," 通常是下句开头（应后向合并），前向合并会让 "Oh." 贴到上一句。**缓解**：安全网只在"无标点结尾"时触发，"Oh," 有逗号不是句末标点但 `SENTENCE_END_RE` 不匹配逗号——需确认安全网是否会把 "Oh," 当半句话合并。**决定**：安全网只看 `.?!。？！`，逗号结尾的句子不被视为"完整句"也不被强制合并（逗号是句内标点，不触发）。已在 spec scenario 覆盖。
- **[gap 计算依赖时间戳]** 安全网用 `sentences[i+1].start - sentences[i].end`，依赖上游时间戳。**缓解**：上游已有 `OVERLAP_BUFFER_SEC=0.3` 截断，时间戳已基本对齐；极端不准时 gap 偏大不合并，不产生错位（最多保留半句话，不比现状差）。
- **[词级 json3 回归]** 安全网对词级数据同样生效。**缓解**：词级数据每词带标点的少，但安全网只在"无标点+gap<1s"时触发，词级数据 gap 通常 >1s（每词独立 event）或已有标点，触发率极低；53 个测试需全跑确认无回归。

## Migration Plan

1. **阶段 1（长句切分标点优先）**：改 `segmentSentences` 长句切分点选择逻辑，跑 53 个测试确认无回归。
2. **阶段 2（安全网）**：在 `segmentSentences` 末尾加安全网循环，跑 53 个测试。
3. **阶段 3（WARN 日志）**：加极端 case 告警。
4. **阶段 4（回归）**：用 `f (2).txt` 跑完整流水线，对比 `subtitle-pipeline-log (3).txt`：
   - #56+#57 合并为一句，不再切开。
   - 24 个无标点结尾 case 降到接近 0。
   - 翻译不错位（#57 起译文与原文对齐）。
5. **回滚**：安全网是独立后处理循环，删除即回滚；长句切分改动局限在切分点选择，恢复"最大间隔"即回滚。

## Open Questions

- 安全网的 `hardBreakBetween(i, i+1)` 如何判定？当前 `_hardBreakAfter` 标记在 segment 级别，安全网阶段 sentences 已是最终对象，标记可能丢失。**倾向**：安全网用 `gap > 2s` 作为硬切的代理判定（>2s 即视为语义断，不合并），不依赖 `_hardBreakAfter` 标记。
- 15s 合并上限是否需要可配置？**倾向**：先硬编码，上线后看日志再决定是否抽常量。
- 安全网是否需要处理"逗号结尾"的句子？英语长句常以逗号连接从句，逗号结尾语义不完整但非半句话。**倾向**：不处理，逗号结尾由 AI 自行断句翻译，安全网只管 `.?!。？！`。
